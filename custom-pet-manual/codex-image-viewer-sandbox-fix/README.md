# Codex 로컬 이미지 뷰어 샌드박스 오류 해결

> 커스텀 Pet 제작 중 발생한 **Codex 로컬 이미지 뷰어 실행 실패**의 진단·해결 기록입니다.
> 이 문서는 특정 PC의 설치법이 아니라, **"로그로 Access Denied가 확인된 경우의 진단 절차"** 로 읽어야 합니다.
> 개인 사진·캐릭터 원본·사용자명·절대 경로 등 민감 정보는 담지 않습니다.

---

## 1. 한눈에 보기

| 항목 | 내용 |
|---|---|
| 증상 | 로컬 이미지 뷰어가 모든 이미지에서 `windows sandbox ... setup refresh had errors`로 실패 |
| 분류 | **검수 도구의 실행 환경 오류** (이미지 생성 품질 문제 아님) |
| 확정 원인 | 작업공간의 쓰기 루트(writable root)에 포함된 `C:\tmp` 폴더의 **ACL 변경 권한 부족** |
| 직접 해결 | 해당 폴더에 샌드박스 그룹이 Modify ACE를 받을 수 있도록 ACL 1회 보정 |
| 효과 미확인 | 선행된 공식 Store `repair`는 이 문제에 효과가 관측되지 않음 |
| 최종 상태 | 동일 작업공간에서 `errors=[]` + 실제 PNG 열림 → **VERIFIED_RESULT** |

---

## 2. 문제 정의

Codex 로컬 이미지 뷰어가 후보 스프라이트 시트·GIF를 열 때 다음으로 실패했습니다.

```text
fs sandbox helper failed with status exit code: 1:
windows sandbox failed: helper_unknown_error: setup refresh had errors
```

그 결과 사람 눈으로 해야 하는 품질 검수(프레임 크기·발 기준선·의상/바지 색·테두리 잔색)가 중단되었습니다.
**이미지 생성 품질과는 별개의, 검수 도구 실행 계층의 장애**로 분류했습니다.

### 사전에 배제한 것
- 이미지 파일 손상 / 형식 오류 → 특정 파일이 아니라 **모든 경로에서** 실패하므로 배제
- 디스크 용량 / 기본 파일 권한 → 정상 확인
- Codex 앱 제품 자체 결함 → 이 사건의 **직접 원인으로는 확인되지 않음**(3장 근거)

---

## 3. 진단 원칙 — 비파괴 우선, 단일 복구

이전 복구 시도에서 **"앱만 종료되고 성공 여부는 확인되지 않는"** 반복이 있었습니다. 이를 막기 위해 다음 순서를 고정했습니다.

1. **증거 수집 (읽기 전용)** — 앱·샌드박스 로그·설치/계정 상태를 무변경으로 점검
2. **복구 적합성 판정** — 로그로 실패 지점을 *설치 / 권한 / 제품 결함* 중 하나로 좁힘
3. **단일 복구** — 원인이 확정된 뒤 **정확히 한 번만** 최소 조치
4. **기능 검증** — 실패하던 동일 조건에서 성공 로그 + 실제 이미지 열림까지 확인
5. **실패 시 종결** — 복구가 실제로 시작되지 않으면 같은 명령을 반복하지 않고 근거를 정리

> 금지: `Reset`, 앱 데이터 삭제, 샌드박스 계정·방화벽·레지스트리 수동 조작, 무증거 반복 재시도.

---

## 4. 근거와 원인 판정

샌드박스 로그(`~/.codex/.sandbox/sandbox.<날짜>.log`)가 실패 지점을 정확히 지목했습니다.

```text
write ACE grant failed on C:\tmp: SetNamedSecurityInfoW failed: 5 (Access denied)
setup refresh: ... errors=["write ACE failed on C:\\tmp: ... failed: 5"]
setup refresh had errors
```

### 확정된 사실
- 이 작업공간을 대상으로 실행될 때만 쓰기 루트에 `C:\tmp`가 포함되었고, 그때만 실패했다.
  (다른 작업 폴더에서는 같은 시각에 `errors=[]`로 성공)
- 샌드박스 설정 과정이 `C:\tmp`에 샌드박스 그룹용 쓰기 ACE를 추가하려다 **Access Denied(5)** 로 실패했다.
- 샌드박스 사용자·그룹 자체는 정상 존재·활성 상태였다 → **설치 계층은 건강**.
- `C:\tmp`의 소유자 컨텍스트에서 사용자 프로세스가 **ACL을 수정할 권한(WRITE_DAC)이 부족**했다.

### 확정하지 않은 사항 (추정과 사실의 분리)
- `C:\tmp`가 **어떤 경위로 생성됐는지**는 증거로 확정되지 않았다. "관리자 권한으로 생성됐다"는 추정이므로 결론에 쓰지 않는다.
- 확정된 것은 **"해당 폴더에서 ACL 변경 권한이 부족했다"** 까지다.

---

## 5. 선행 조치의 효과 판정 (정직 기록)

- 이 사건 이전에 공식 Microsoft Store `repair`가 **1회 실행**되었다.
- 그 뒤에도 이미지 뷰어 오류는 **그대로 남았다.**
- 따라서 **Store Repair가 이 문제를 해결했다는 근거는 없으며, 관측상 효과는 확인되지 않았다.**
- 교훈: Store Repair를 만능 해결책으로 기록하지 않는다. 로그가 권한 실패를 가리킬 때는 재설치·복구 반복보다 **권한 계층을 먼저** 본다.

---

## 6. 직접 수정 (단일·가역·앱 종료 없음)

로그가 가리킨 대상 폴더에 대해 **승격된 `icacls`를 한 번** 실행해, 샌드박스가 필요로 하는 ACE를 받을 수 있도록 ACL을 보정했습니다.

```powershell
# "로그로 write ACE 실패(Access Denied)가 확인된 경우"에 한한 진단적 조치.
# <TARGET_ROOT> = 로그의 "write ACE grant failed on ..." 에 나온 폴더 (예: C:\tmp)
# <USER>        = 현재 사용자 계정
icacls "<TARGET_ROOT>" /grant "<USER>:(OI)(CI)F"
icacls "<TARGET_ROOT>" /grant "CodexSandboxUsers:(OI)(CI)M"
```

- 수정 중 **파일 삭제·앱 강제 종료·샌드박스 계정 재생성·레지스트리/방화벽 변경·Reset·재설치는 하지 않았습니다.**
- 조치는 정확히 **1회**, 원인이 확정된 뒤에만 실행했습니다.

> ⚠️ 이 명령은 일반 설치법이 아닙니다. 로그로 특정 폴더의 ACL 권한 실패가 확인됐을 때만 적용합니다.

---

## 7. 검증 결과 (VERIFIED_RESULT)

실패하던 **동일 작업공간·동일 쓰기 루트 조건**에서 성공이 관측되었습니다.

```text
setup refresh: processed 3 write roots (read roots delegated); errors=[]
```

- 로그 기준: `errors=[]` (이전엔 같은 조건에서 항상 실패)
- 사용자 체감 기준: 후보 접촉 시트 PNG가 **실제로 열림**

두 기준이 모두 통과하여 이 장애는 **닫힌 사건**입니다.

---

## 8. 금지된 롤백과 재발 방지

### 되돌리지 말 것
아래는 현재 정상 상태를 제거해 **같은 오류를 재발시키므로 실행 금지**입니다.

```powershell
icacls "<TARGET_ROOT>" /remove "<USER>" "CodexSandboxUsers"   # 실행 금지
```

### 새 writable root 추가 시 사전 점검 (체크리스트)
1. 해당 폴더에서 `CodexSandboxUsers`가 **Modify ACE를 받을 수 있는 ACL**인가
2. 현재 실행 컨텍스트가 필요한 **ACL 변경 권한**을 갖는가
3. 설정 후 동일 작업공간에서 `setup refresh`가 `errors=[]`인가
4. **실제 이미지 하나를 열어** 사용자 체감 결과까지 확인했는가

---

## 9. 핵심 교훈

- **도구 결함과 자산 결함을 분리한다.** 이미지 뷰어가 안 열리는 것은 Pet 그림 품질과 무관할 수 있다.
- **로그가 실패 지점을 지목한다.** 앱 재시작·재설치를 반복하기 전에 `~/.codex/.sandbox/`의 로그부터 읽는다.
- **복구는 원인 확정 후 1회.** 성공 로그와 실제 열림까지 확인돼야 "수정 완료"다.
- **사실·추정·미확인을 섞지 않는다.** 원인 폴더의 생성 경위처럼 확정 안 된 것은 결론에 넣지 않는다.
